# “安排”模块设计标准

## 设计方向

“安排”模块是一个安静的未来意图工作台。它应该像一本会自己整理线索的随身手账，而不是一个不断催促用户的绩效面板。

Domain：

- 未来意图
- 时间线
- 稍后
- 上下文
- 轻确认
- 日历总览
- 完成与放下

Color world：

- 晨雾灰：用于基础背景和弱分组。
- 纸面白：用于主内容。
- 墨色黑：用于主要文字。
- 苔绿：用于主要操作和完成反馈。
- 麦穗浅金：用于温和提示，不用于警告。
- 暖石灰：用于“时间过去了”和“稍后”状态。

Signature：

“以后再说”是本模块的签名交互。它不是二级菜单里的清理动作，而是和完成同级的低压力出口。

Rejecting：

- 拒绝红色逾期列表，改为“时间过去了”的柔和整理区。
- 拒绝卡片海，改为清晰的时间分组和列表-详情节奏。
- 拒绝花哨动效，改为短、稳、可中断的状态反馈。

## 视觉规则

1. 不使用红色表达逾期。
2. 颜色只表达状态和操作，不做装饰。
3. 卡片圆角保持克制，默认不超过 8px。
4. 信息密度要适合移动端扫读，每条安排先露出标题、时间、人/地点摘要、状态。
5. 无时间安排和“以后再说”安排要可见，但不抢占今天优先级。

## 交互规则

1. 创建入口常驻，但不遮挡内容。
2. 完成可以通过勾选或右滑完成。
3. “以后再说”可以通过右滑或长按出现。
4. 到期未完成时，不弹阻断式提醒。
5. 批量整理入口只用温和文案，例如：“有 3 件事时间过去了，要整理一下吗？”

## 动效规则

1. 高频操作使用 100ms 到 160ms 的反馈。
2. 面板、详情页、日期切换使用 200ms 到 250ms。
3. 只动画 `transform`、`opacity`、颜色，不动画布局属性。
4. 右滑操作必须跟随手指，释放后再使用缓动。
5. 遵守 `prefers-reduced-motion`，减少空间位移。

## 可访问性规则

1. 所有可点击区域使用 `button` 或具备等效键盘行为。
2. 勾选、稍后、删除、关闭等图标按钮必须有明确 `aria-label`。
3. 状态不能只靠颜色表达，必须有文字或图标辅助。
4. 表单字段必须有 label。
5. 日历中当天、选中、包含安排的日期必须有非颜色提示。
