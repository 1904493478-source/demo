# Arrangements V0.4 Intelligent Completion Plan

## 阶段状态

当前 V0.4 最小闭环已完成可收尾。收尾验证时间：2026-05-18 04:10 CST。

本阶段稳定边界是：智能完成只进入 `可能已完成` 确认态；无 @ 群聊只生成低置信待确认候选；正式安排列表和日历已补齐低压整理能力，包括已完成折叠、日历按日切换、重复安排去重和已完成卡片左滑删除。

## 阶段目标

V0.4 先做智能完成的最小闭环，再补无 @ 群聊低置信候选。原则是：AI 和本地识别只能提出建议，不能静默替用户完成安排。

## 已实现切片

### Slice 1: 可能已完成

- 后续聊天里出现明确完成语义，例如 `我已经去公司拿到资料了`、`我今天上午已经去医院体检了`，会匹配当前未完成安排。
- 匹配成功后安排状态变为 `可能已完成`，`completedAt` 保持为空。
- `可能已完成` 会放在今天板块，方便用户轻量确认。
- 详情页显示低压绿色提示，并提供 `确认完成` 和 `还没完`。
- 用户点 `确认完成` 后才进入已完成；点 `还没完` 会回到原来的时间分组。

### Slice 2: 无 @ 群聊低置信候选

- 群聊中没有 @ 当前用户，但出现开放请求，例如 `谁可以帮我去公司拿资料？`。
- 只有当前用户后续明确回复一人称承诺，例如 `可以，我去拿`，才生成候选。
- 候选 `confidence` 为 `low`，仍只进入待确认候选，不直接创建正式安排。
- 明确 @ 当前用户的群聊请求继续保持原有更高置信路径。

### Slice 3: 收尾稳定性

- 候选确认生成的正式安排，详情主按钮显示 `确认安排`，不会误触发已完成。
- 已完成分组默认只展示最近完成的 3 条，更多内容通过 `展开更多` 查看，避免页面无止境变长。
- 列表所有分组保留同一套左滑删除和二次确认；已完成卡片也支持从整张卡片表面左滑露出删除。
- 日历总览默认只展示当天安排，点击某一天后只显示该天安排。
- 安排数据在读取、保存和页面状态更新时按 `id` 去重，避免重复候选确认造成日历堆叠和多卡片联动左滑。

## 暂不包含

- 使用大模型判断完成状态。
- 自动完成安排。
- 无 @ 群聊中多人分工、多人抢答、撤回和否定修正。
- 正式安排之间的自动合并。
- 日历月视图、跨月切换和系统通知。

## 验证场景

- 创建或保留一条 `明天下午帮用户A去公司拿资料`，在聊天中发送 `我已经去公司拿到资料了`，安排页显示 `可能已完成`，但不进入已完成。
- 打开这条安排详情，点击 `确认完成` 后才进入已完成。
- 打开这条安排详情，点击 `还没完` 后回到今天或近期分组。
- 群聊消息 `明天下午3点谁可以帮我去公司拿资料？`，用户回复 `可以，我去拿`，安排页出现低置信 `groupChat` 候选。
- 在日历中来回点击两个日期，列表不应累加重复内容。
- 已完成安排超过 3 条时默认只显示最近 3 条；展开后较早完成的安排仍可左滑删除。
- 对重复 `id` 的历史本地数据，页面只展示一条，且左滑一个安排不会带动其他卡片。

## 下一阶段建议入口

- 优先做智能完成的模型辅助匹配：在本地规则命中不足时调用用户配置的 API，但仍只进入 `可能已完成`。
- 为无 @ 群聊候选增加置信解释和“不是我的事”反馈入口，降低误识别成本。
- 设计正式安排合并后的拆分、撤销和来源冲突处理，再进入正式安排自动合并。
