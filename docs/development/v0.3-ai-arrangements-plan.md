# Arrangements V0.3 Group Chat Recognition Plan

## 阶段状态

当前 V0.3 已完成可进入下一阶段。收尾验证时间：2026-05-18 03:01 CST。

V0.3 的稳定边界是：群聊 @ 当前用户请求、AI 精修、候选层上下文归集和场景合并。所有识别结果仍只进入待确认候选，不直接创建正式安排。

## 目标

V0.3 从群聊开始扩展 AI 安排识别，但继续遵循最小可用原则：先只识别和当前用户明确相关的群聊请求，生成待确认候选安排，不做自动入库、不做群成员同步、不做复杂归集合并。

这一版验证的问题是：群聊里信息更嘈杂时，用户是否仍能接受“隐式识别 + 候选确认”的低打扰方式。

## 当前 Slice: 群聊 @我 请求

### 已实现范围

- 群聊中其他成员发送明确指向当前用户的请求，例如 `@李小溪 明天下午3点帮我去公司拿资料`。
- 当前用户回复接受或承诺，例如 `可以，我去拿`。
- 系统生成一条 `groupChat` 待确认候选安排。
- 候选仍需用户在“安排”页确认、编辑或忽略。
- 原群聊请求气泡出现低存在感识别反馈，不显示显式弹窗文案。
- 历史已接受群聊消息可在页面刷新后回填候选和反馈。

### 本切片新增

- 新发送的群聊回复会在本地候选生成后，使用用户已配置的 OpenAI-compatible API 做一次 AI 精修。
- AI 精修结果只覆盖待确认候选，不直接创建正式安排。
- API 不可用、未配置或模型返回不创建时，保留本地候选兜底。
- 历史群聊回填仍只使用本地识别，避免打开页面就消耗 token。
- 同主题待确认候选会归集成一条，例如爸爸和姐姐都提醒去医院时，只保留一个医院候选。
- 合并逻辑按“场景”判断：明确同主题可合并；或同一地点且时间相隔不超过 3 小时也合并。
- 合并后的候选详情保留所有原始对话上下文，并显示每个来源的可信度。
- 多个来源给出不同时间时，候选卡片显示“时间有出入”，确认成正式安排后备注中继续保留冲突提示。

### 暂不包含

- 未 @ 当前用户但语义上可能相关的群聊识别。
- 群聊里多人分别承担不同事项。
- 群聊安排同步给对方或其他成员。
- 正式安排之间的跨安排自动合并。

## 判断逻辑

候选生成必须同时满足：

- 消息来自群聊成员。
- 请求文本明确包含当前用户名称的 `@` 提及。
- 文本中包含可执行目标。
- 当前用户后续回复表达接受或承诺。

不满足以上条件时不生成候选，继续保留手动创建入口。

## 验证场景

- 在测试群中发送 `@李小溪 明天下午3点帮我去公司拿资料`，用户回复 `可以，我去拿`，应生成 `明天下午3点帮用户A去公司拿资料` 候选。
- 原群消息下方应出现轻量识别反馈，不能出现“已识别并添加候选”一类显式弹窗。
- 进入“安排”页，候选卡片来源显示 `群聊`，仍可确认、编辑、忽略。
- 如果群聊消息 @ 的不是当前用户，即使用户回复 `可以`，也不生成候选。
- 配置 AI API 后，新的群聊回复会调用 `/chat/completions` 并用模型标题精修候选。
- 爸爸和姐姐分别提醒去医院时，候选区只出现一条医院候选，卡片显示 `2 个来源`、两段原始对话和对应可信度。
- 如果两个来源的时间不一致，卡片显示 `时间有出入`，确认后正式安排备注仍保留所有来源与冲突提示。
- 同一地点且时间相近的候选，例如上午在市中心医院挂号和取报告，会归为一个场景候选；同地点但时间相隔过大则保持分开。

## 后续切片建议

- Slice 2：群聊候选 API 精修，只在新回复时调用用户配置的 API，历史回填仍保持本地。已完成。
- Slice 3：无 @ 但包含当前用户昵称或“你”的上下文识别，建议进入下一阶段作为低置信候选能力。
- Slice 4：同主题候选归集合并，详情保留所有原始对话。已完成候选层最小版。
- Slice 5：智能完成的最小场景，例如用户后续说“我已经拿到资料了”，建议作为下一阶段核心目标。

## 下一阶段建议入口

- 优先做智能完成的最小闭环：当用户后续明确说“已经去了/拿到了/买好了”时，给对应安排进入“可确认完成”状态，而不是直接强制完成。
- 其次补无 @ 群聊相关性：仅生成低置信候选，避免群聊噪声造成误识别。
- 暂缓正式安排自动合并：目前只在候选层合并，正式安排合并需要更强撤销和拆分能力。
